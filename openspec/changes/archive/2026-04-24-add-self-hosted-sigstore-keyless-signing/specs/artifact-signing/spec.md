## ADDED Requirements
### Requirement: Release pipelines SHALL support keyless OCI signing against self-hosted Sigstore
ServiceRadar release and OCI publish pipelines SHALL support keyless Cosign signing against operator-configured Sigstore components rather than requiring a static Cosign private key in CI.

#### Scenario: Forgejo release workflow signs published OCI images keylessly
- **GIVEN** Forgejo Actions OIDC is enabled for the release job
- **AND** custom Fulcio and Rekor endpoints are configured for the pipeline
- **WHEN** the release workflow publishes OCI images
- **THEN** the workflow SHALL sign the published images without a static Cosign private key secret
- **AND** the resulting signatures SHALL be visible to Harbor and verifiable by Cosign

#### Scenario: BuildBuddy release helper signs with the same custom Sigstore configuration
- **GIVEN** the BuildBuddy release helper is used to publish OCI images
- **WHEN** image signing is executed
- **THEN** it SHALL support the same custom Sigstore endpoint and trust configuration as Forgejo release automation
- **AND** verification SHALL occur only after signing succeeds

### Requirement: Keyless signing trust SHALL be bound to an approved OIDC issuer and workflow identity
The signing system SHALL bind keyless certificates to an explicitly approved OIDC issuer and constrained workflow identity claims.

#### Scenario: Unapproved issuer is rejected
- **GIVEN** a workflow presents an OIDC token from an issuer not configured for ServiceRadar signing
- **WHEN** Fulcio or the signing policy evaluates the token
- **THEN** signing SHALL be rejected
- **AND** no signature SHALL be published for the OCI artifact

#### Scenario: Workflow identity claims are constrained
- **GIVEN** a workflow requests a signing certificate
- **WHEN** the token claims are evaluated
- **THEN** the approved repository, ref, and workflow identity constraints SHALL be enforced
- **AND** tokens outside those constraints SHALL NOT be accepted for release signing

### Requirement: Custom Sigstore trust material SHALL be published for verification
The system SHALL publish the trust material required to verify keyless signatures generated by the self-hosted Sigstore components.

#### Scenario: Operator verifies a published release image locally
- **GIVEN** an operator has access to the published trust root and a released OCI image reference
- **WHEN** they run Cosign verification against the release image
- **THEN** verification SHALL succeed without requiring hidden CI-local trust state
- **AND** the verification flow SHALL use the published custom trust material

#### Scenario: Verification fails against stale or mismatched trust material
- **GIVEN** verification is attempted with trust material that does not match the active self-hosted Sigstore deployment
- **WHEN** Cosign verifies the OCI signature
- **THEN** verification SHALL fail clearly
- **AND** the failure SHALL indicate a trust-root or issuer mismatch rather than silently passing
