# Design: add-falco-nats-integration

## Context

ServiceRadar uses NATS JetStream as its ingestion backbone. All collectors authenticate to NATS
using JWT credentials generated by the existing onboarding pipeline:

```
CollectorPackage (Ash resource)
  -> ProvisionCollectorWorker (Oban)
    -> AccountClient.generate_user_credentials() (gRPC to datasvc)
      -> accounts.GenerateUserCredentials() (Go: NKey + JWT signing)
        -> .creds file content returned
  -> AshCloak encrypts and stores in DB
  -> Bundle download: tarball with creds + certs + config
  -> CLI enrollment: installs to /etc/serviceradar/
```

Falcosidekick is patched to support `.creds` file auth and custom CA cert paths in its NATS output,
enabling direct connection to ServiceRadar's NATS JetStream.

## Upstream Falcosidekick Changes

### New NATS Output Configuration Fields

Added to Falcosidekick's NATS output:

| Setting             | Env var              | Default | Description                                        |
|---------------------|----------------------|---------|----------------------------------------------------|
| `nats.credsfile`    | `NATS_CREDSFILE`     | `""`    | Path to NATS `.creds` file for JWT authentication  |
| `nats.cacertfile`   | `NATS_CACERTFILE`    | `""`    | Path to CA certificate for server TLS verification |

When `credsfile` is set, the NATS client uses `nats.UserCredentials(credsfile)` option.
When `cacertfile` is set, it is added to the TLS `RootCAs` pool.
These work alongside the existing `mutualtls` and `checkcert` settings.

## Onboarding Integration

### New Collector Type

Add `:falcosidekick` to `ProvisionCollectorWorker.build_permissions_for_collector/1`:

```elixir
defp build_permissions_for_collector(:falcosidekick) do
  %{
    publish_allow: ["events.falco.>"],
    subscribe_allow: ["_INBOX.>"],
    response_max: 1
  }
end
```

### Bundle Contents

The `CollectorBundleGenerator` produces a tarball with:

```
falcosidekick-bundle.tar.gz
+-- creds/
|   +-- nats.creds          # NATS JWT credentials
+-- certs/
|   +-- ca-chain.pem        # NATS CA cert for TLS verification
+-- README.md               # Quick-start instructions
```

Note: no config file in the bundle. Falcosidekick is configured via its own Helm values
or environment variables pointing to the mounted creds/certs paths.

### Enrollment

Same as existing collectors:
1. Admin creates `falcosidekick` package via UI or API
2. Gets enrollment token
3. On target cluster: download bundle, create K8s Secret from creds + certs
4. Configure Falcosidekick Helm values to mount the secret and set NATS env vars

## Deployment

### Helm Values for Falcosidekick

Users add to their Falcosidekick Helm values:

```yaml
falcosidekick:
  config:
    nats:
      hostport: "nats://nats.serviceradar.example.com:4222"
      subjecttemplate: "events.falco.raw"
      minimumpriority: "warning"
      checkcert: true
      cacertfile: "/etc/serviceradar/certs/ca-chain.pem"
      credsfile: "/etc/serviceradar/creds/nats.creds"

  extraVolumes:
    - name: serviceradar-nats
      secret:
        secretName: falcosidekick-serviceradar-creds

  extraVolumeMounts:
    - name: serviceradar-nats
      mountPath: /etc/serviceradar
      readOnly: true
```

The K8s Secret is created from the downloaded bundle:

```bash
kubectl create secret generic falcosidekick-serviceradar-creds \
  --from-file=creds/nats.creds=creds/nats.creds \
  --from-file=certs/ca-chain.pem=certs/ca-chain.pem \
  -n falco
```

## NATS Stream Compatibility

The existing `events` JetStream stream includes `events.>` as a subject.
`events.falco.raw` is captured by this wildcard — no stream configuration changes needed.

## Security Considerations

- NATS credentials are scoped: can only publish to `events.falco.>`, subscribe to `_INBOX.>`
- Each cluster gets unique NKey identity via the onboarding pipeline
- Credentials can be revoked per-cluster without affecting others
- TLS transport to NATS with CA verification
- Secrets mounted read-only from K8s Secrets
- No NATS server config changes per Falcosidekick instance (JWT resolver handles auth)

## Message Format

Falcosidekick sends events as JSON. The raw payload is published to NATS as-is.
Downstream consumers (future db-event-writer processor) will handle parsing.

Example Falco event:
```json
{
  "uuid": "...",
  "output": "Unexpected process spawned in container (user=root ...)",
  "priority": "Warning",
  "rule": "Terminal shell in container",
  "time": "2026-03-03T04:30:00.000000000Z",
  "output_fields": {
    "container.id": "abc123",
    "container.name": "myapp",
    "k8s.ns.name": "production",
    "k8s.pod.name": "myapp-7f8b9c-x2k4p",
    "proc.cmdline": "bash",
    "user.name": "root"
  },
  "source": "syscall",
  "tags": ["container", "shell", "mitre_execution"],
  "hostname": "worker-node-3"
}
```
