## Context
Current plugin import requires multiple uploads and embeds `display_contract` in YAML. This is error-prone and unfriendly to plugin authors who expect a single artifact that can be shared and versioned.

## Goals / Non-Goals
- Goals:
  - Standardize a bundle layout that can be generated by templates and SDK examples.
  - Allow sidecar JSON files for display contracts and config schemas.
  - Keep compatibility with existing non-bundled uploads.
- Non-Goals:
  - Implement SDKs (tracked in #2503 / #2504).
  - Introduce new runtime features or host functions.

## Decisions
- **Bundle format**: Use a `.zip` file with a predictable layout:
  - Required: `plugin.yaml`, `plugin.wasm`
  - Optional: `config.schema.json`, `display_contract.json`, `README.md`, `LICENSE`
- **Precedence**:
  - If `display_contract.json` is present, it overrides any `display_contract` embedded in `plugin.yaml`.
  - If `config.schema.json` is present, it is used as the config schema; manifest-embedded schemas (if ever added) are ignored.
- **Validation**:
  - Bundle validation returns actionable errors (missing files, invalid schema JSON, invalid manifest fields).
  - Bundle upload flows into the existing staged approval process.

## Risks / Trade-offs
- Users may still upload separate files; we must preserve both paths for backward compatibility.
- Zip extraction must guard against path traversal; only accept files at the root and known filenames.

## Migration Plan
- Add bundle support without removing existing file-by-file uploads.
- Update examples and docs first to guide new users toward bundle uploads.

## Open Questions
- Should we allow a `checksums.txt` or signature file inside the bundle for offline verification?
- Do we want to enforce a maximum bundle size beyond existing Wasm size limits?
